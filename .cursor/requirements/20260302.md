# 需求描述文档：高性能用户系统 (User System)

## 1. 项目概述
构建一个生产级的用户中心模块，支持高并发环境下的用户注册、登录与退出功能。系统需具备健壮的错误处理、安全的密码存储以及完整的自动化测试覆盖。

## 2. 核心功能需求

### 2.1 用户注册 (Register)
- 输入: 用户名 (Username, 唯一)、密码 (Password, 至少8位)、邮箱 (Email, 可选)。
- 逻辑:
    - 检查用户名是否已存在（高并发下需保证数据库唯一性约束）。
    - 使用 bcrypt 对密码进行加密哈希存储。
    - 注册成功后返回用户基本信息（不含密码）。
- 并发要求: 数据库连接池需优化，防止在高频请求下出现连接枯竭。

### 2.2 用户登录 (Login)
- 输入: 用户名、密码。
- 逻辑:
    - 验证用户名是否存在。
    - 比对加密哈希密码。
    - Session 管理: 登录成功后生成 Token (JWT 或 UUID)，并将其存入 Redis，设置过期时间（如 2 小时）。
    - DB中还需要有一个字段：is_login 和 login_at 来标记用户此时是否已经登录和登录时间，便于统计和避免用户多设备登录；
    - 返回 Token 给客户端。

### 2.3 用户退出 (Logout)
- 输入: Authorization Header (Token)。
- 逻辑:
    - 从 Redis 中删除对应的 Token。
    - 确保该 Token 立即失效。

## 3. 非功能性需求 (高并发与安全)

### 3.1 数据库规范 (MySQL)
- 表结构: users 表必须包含 id (bigint), username (varchar), password_hash (string), created_at, updated_at。
- 索引: 对 username 建立唯一索引。
- 连接池: 在 Go 中配置 MaxOpenConns=100, MaxIdleConns=20。

### 3.2 缓存规范 (Redis)
- 使用 Redis 存储 Session/Token 以支撑横向扩展。
- Key 格式建议: user:session:{token}。

### 3.3 安全规范
- 严禁在代码或日志中打印明文密码。
- 接口需具备基础的频率限制 (Rate Limiting) 意识（可在 Handler 层预留钩子）。

## 4. 交付物要求

### 4.1 代码实现 (Go)
- 按照 .cursor/rules/100-go-conventions.mdc 的分层架构实现。
- 提供 cmd/main.go 作为入口。
- 架构模式: 必须采用 DDD (领域驱动设计)，实现严格的四层解耦，确保领域逻辑（Domain）不依赖外部框架。
- 提供sql生成语句，存在docs/db.sql中，内容是服务接口对应的数据库表的建表语句，包括索引，和comment；

### 4.2 接口文档 (Markdown)
- 自动生成 docs/api.md，必须包含 Request Body 示例和 Response Code 说明。

### 4.3 自动化测试 (Pytest)
- 在 tests/ 编写黑盒测试脚本。
- 测试场景:
    - 正常注册 -> 正常登录 -> 携带 Token 退出。
    - 使用错误密码登录（预期 401）。
    - 注册已存在的用户名（预期 409）。
